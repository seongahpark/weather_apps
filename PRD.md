# 날씨 기반 옷차림/대비/점심 메뉴 추천 서비스 (Python + Streamlit)

## 1. 문서 목적
본 문서는 **실시간 날씨 정보를 기반으로** 사용자의 하루 의사결정(옷차림, 외출 대비, 점심 메뉴)을 돕는 **Streamlit 웹앱**의 요구사항을 정의한다.
날씨 데이터는 OpenWeatherMap을 기본으로 사용하며, 추천 문구 생성은 규칙 기반(Deterministic)과 LLM 기반(Generative)을 모두 지원하는 것을 목표로 한다. (OpenWeatherMap 호출 및 날씨 정보 함수화 예시는 업로드된 메모에 포함되어 있음 )

## 2. 제품 개요
### 2.1 문제 정의
* 사용자는 “오늘 뭐 입지?”, “우산 챙겨야 하나?”, “점심 뭐 먹지?”를 날씨와 함께 반복적으로 고민한다.
* 날씨 앱은 정보를 제공하지만 **행동 지침(Outfit/Prep/Food)**까지 일관되게 연결해주는 경험은 부족하다.

### 2.2 해결 개념
* 사용자의 위치/관심 지역의 **현재 날씨 및 (옵션)단기 예보**를 가져온 뒤,
* 아래 3종 추천을 **한 화면에서** 제공한다.

  1. 옷차림 추천
  2. 대비/준비물 추천(우산, 미세먼지 마스크, 핫팩 등)
  3. 점심 메뉴 추천(기온·강수·습도·체감 기반)

## 3. 목표(Goals) / 비목표(Non-Goals)
### 3.1 목표
* 사용자가 10초 내에 “오늘의 행동”을 결정할 수 있는 **요약 중심 UX**
* 날씨 신호(기온/체감/강수/풍속/습도)를 **일관된 규칙**으로 해석하여 추천 품질 안정화
* LLM 사용 시에도 **사실(날씨 데이터)과 추천(생성 문구)을 분리**해 신뢰성 확보
### 3.2 비목표
* 장기(7~14일) 고정밀 예측 모델 구축
* 배달/예약/결제 등 트랜잭션 기능
* 의료적/치료적 조언(알레르기 질환 등) 제공

## 4. 대상 사용자(Personas)
1. **출근/외근 직장인**: 빠른 결정을 원함(우산·외투·구두 등)
2. **학생/프리랜서**: 하루 활동(산책/운동/카페)과 연동된 추천 선호
3. **가족 단위 사용자**: 아이 옷차림/외출 준비물 체크리스트 선호

## 5. 사용자 시나리오 & 유저 스토리
### 5.1 핵심 시나리오
* 사용자는 지역을 선택한다(기본값: Seoul).
* 앱은 현재 날씨(기온/습도/바람/강수/상태)를 표시한다.
* 앱은 “오늘의 추천” 3종을 카드 형태로 제공한다.
* 사용자는 필요 시 “아침/점심/저녁(옵션: 예보 기반)” 탭으로 전환한다.

### 5.2 유저 스토리(예시)
* (옷차림) “현재 2°C, 바람 강함이면 패딩+목도리+장갑 추천을 받고 싶다.”
* (대비) “강수 확률 높으면 우산/방수 신발을 안내받고 싶다.”
* (점심) “덥고 습하면 시원한 메뉴, 추우면 국물 메뉴를 제안받고 싶다.”
* (신뢰) “추천의 근거(기온/강수/풍속)를 간단히 확인하고 싶다.”

## 6. 기능 요구사항
### 6.1 날씨 조회
**필수**
* OpenWeatherMap 현재 날씨 API 연동 (도시명 또는 위경도)
* 응답 파싱: 날씨 설명, 기온(섭씨), 습도, 풍속(가능하면), 강수(가능하면)
* 실패 처리: API 에러/타임아웃 시 사용자에게 메시지 + 재시도 버튼

**권장**
* 단기 예보(3시간 단위/일 단위) 추가하여 시간대별 추천 제공
* Streamlit 캐시 적용(예: 5~10분 TTL)
> 참고: 업로드된 메모에는 OpenWeatherMap 호출 예시 및 `weather_info(city)` 함수 형태가 포함되어 있다. 

### 6.2 추천 엔진(Outfit / Prep / Lunch)
추천은 **2가지 모드**로 동작한다.

#### A) 규칙 기반(기본)
* 입력: 기온, 체감(가능하면), 습도, 풍속, 강수(가능하면), 날씨 상태(description)
* 출력: 구조화된 추천(JSON-like dict) + 사용자용 문장
* 장점: 일관성/테스트 용이/비용 0

**예시 룰(요약)**
* 기온:
  * ≥ 28°C: 반팔/린넨/통풍, 냉음식/면류 추천
  * 23~27°C: 얇은 셔츠/가디건, 샐러드/덮밥/국수
  * 17~22°C: 긴팔/얇은 자켓, 따뜻한 국/비빔밥
  * 10~16°C: 자켓/가디건+레이어드, 국밥/찌개
  * 0~9°C: 코트/패딩, 장갑/목도리, 국물/전골
  * < 0°C: 방한 강화(내복/핫팩), 뜨거운 국물/탕
* 강수/비:
  * 비/눈/소나기: 우산, 방수 아우터, 미끄럼 주의
* 바람:
  * 풍속 강함: 체감온도 하락 → 방풍 외투 권장
* 습도:
  * 고습: 통풍 소재/불쾌지수 안내, 시원한 메뉴
  * 저습: 보습/정전기/립밤 등 간단 안내(과도한 건강 조언 금지)

#### B) LLM 기반(옵션)
* 규칙 기반이 만든 **구조화 추천**을 바탕으로 “자연어 표현”만 생성
* 입력 프롬프트에는 **사실(날씨 데이터) + 구조화 추천 결과**를 포함
* 출력은 “최종 사용자 문구”로 제한

> 업로드된 메모에는 OpenAI 클라이언트를 사용한 `recommand_clothes(weather, temp, humidity)` 예시가 포함되어 있다. 
> 단, 제품화 시에는 **API 키를 코드에 하드코딩하지 않고 환경변수로 관리**한다(보안 요구사항 참조).

### 6.3 UI/UX (Streamlit)
**레이아웃(권장)**
* 상단: 위치 선택(도시 검색/드롭다운) + “새로고침”
* 중단: 현재 날씨 요약 카드(기온/체감/습도/바람/강수/아이콘)
* 하단: 추천 3종 카드
  * 옷차림(상·하의/아우터/신발/소품)
  * 대비 체크리스트(우산/마스크/핫팩 등)
  * 점심 메뉴(3~5개, “왜 이 메뉴인지” 1줄 근거)

**부가 요소**
* “추천 근거 보기” 토글: 사용한 주요 신호(기온/강수/풍속 등) 노출
* “모드 선택”: 규칙 기반 / LLM 문장 강화(옵션)

## 8. 데이터/인터페이스 설계
### 8.1 내부 데이터 모델(예시)
* `WeatherObservation`
  * city, lat, lon, observed_at
  * temp_c, feels_like_c(옵션), humidity, wind_speed(옵션)
  * precipitation_mm(옵션), condition, description
* `Recommendation`

  * outfit: items[], rationale
  * prep: checklist[], rationale
  * lunch: options[], rationale
  * confidence / rule_hits(디버그용)

### 8.2 설정/환경변수
* `OPENWEATHER_API_KEY`
* (옵션) `OPENAI_API_KEY`
* `DEFAULT_CITY=Seoul`
* `CACHE_TTL_SECONDS=600`

## 9. 보안/프라이버시
* API 키 하드코딩 금지(환경변수/Secrets 관리)
* 위치 정보:
  * 사용자가 위경도를 직접 입력하는 경우, 저장하지 않음(세션 내 처리)
* 로깅:
  * 개인 식별 가능 정보(정확한 위치/좌표) 기본 미로그 또는 마스킹

## 10. 관측/지표(Analytics)
* DAU/WAU, 추천 카드 클릭(근거 보기 토글 포함)
* 도시 선택 빈도 Top N
* LLM 모드 사용률 및 호출당 비용(추정)
* 오류율(OpenWeather API 실패율, 타임아웃)

## 11. QA / 테스트 전략
* 단위 테스트:
  * 날씨 파서(필수 필드 누락/타입 변화 대응)
  * 추천 룰(경계값: 9°C↔10°C 등)
* 통합 테스트:

  * API 호출 실패/레이트리밋 시 UX 확인
* 회귀 테스트:

  * 룰 변경 시 과거 케이스 스냅샷 비교
